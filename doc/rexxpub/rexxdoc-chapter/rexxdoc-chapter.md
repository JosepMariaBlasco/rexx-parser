# The rexxdoc-chapter Document Class

This document is both a demonstration and a guide to the
`rexxdoc-chapter` document class in RexxPub.  You are reading a
Markdown file rendered through the same class it documents --- so
everything you see on the page is a live example of the features
being described.

The `rexxdoc-chapter` class replicates the visual style of the
official ooRexx documentation, which is generated by DocBook XSL
Stylesheets with Apache FOP.  It is designed for two main scenarios:

- **Online rendering** via the CGI pipeline, for individual
  chapters that are fast enough to display in a browser with
  Rexx syntax highlighting.
- **PDF generation** via `md2pdf`, for chapters or standalone
  technical documents.
- **EPUB generation** via `md2epub`, for ebook versions of
  chapters or standalone technical documents.

## Design Goals

The primary goal is fidelity to the official ooRexx documentation
style: a technical reader who is already familiar with the ooRexx
manuals should feel immediately at home.

A secondary goal is **minimal friction**: any existing Markdown file
can be turned into a rexxdoc chapter by adding four lines of YAML
front matter.  No structural changes to the Markdown are needed.

### Typography

Unlike the `article` and `book` classes, which follow LaTeX
conventions (Times New Roman, justified text, first-line indent),
`rexxdoc-chapter` uses the same sans-serif typography as the ooRexx
manuals:

- **Body text:** Liberation Sans at 10pt, left-aligned (no
  justification, no hyphenation).
- **Code blocks:** Liberation Mono at 8pt, on a light grey
  background (`#f4f5f4`).
- **Headings:** Liberation Sans Bold, in blue (`#336699`).
- **Links:** blue (`#0066cc`), no underline.

Paragraphs are separated by vertical space (DocBook style), with no
first-line indent.

### Heading Hierarchy

The `rexxdoc-chapter` class shifts the heading hierarchy by one level
compared to other RexxPub classes:

- `#` (h1) --- the **chapter title**.  A "Chapter N." label is
  generated automatically via CSS `::before`, with a thin horizontal
  rule below the label.
- `##` (h2) --- **section** headings.  Numbered N.1, N.2, etc.
- `###` (h3) --- **subsection** headings.  Numbered N.1.1, etc.
- `####` (h4) --- **sub-subsection** headings.  Numbered N.1.1.1, etc.

This means that any Markdown file with the standard `#`/`##`/`###`
structure can be used as a chapter with no changes to the source ---
just add the YAML front matter.

## YAML Front Matter

A typical YAML front matter block for a chapter looks like this:

```
---
rexxpub:
  docclass: rexxdoc-chapter
  chapter: 7
---
```

If the file is named `rexxdoc-chapter.md`, the `docclass` can be omitted
(it will be inferred from the filename). The `chapter` value defaults to 1
if not specified. It determines both the "Chapter N." label above the
title and the prefix for all section numbers (7.1, 7.2, 7.1.1, etc.).

Section numbering is always enabled at depth 4 (the default for the
ooRexx documentation) and cannot be overridden --- this is a fixed
design decision to match the official documentation style.

If no `chapter` value is provided, the document defaults to chapter 1.

## Code Blocks and Syntax Highlighting

Because the Rexx Highlighter is the standard mechanism for code
blocks in RexxPub, the `rexxdoc-chapter` CSS only controls the
container styling (background, padding, margins).  The actual syntax
colouring is handled by the active highlighting theme.

For example, this is a Rexx-highlighted fenced code block:

```rexx {unicode}
::Method open Package Protected         -- Bold, underline, italic
  Expose x pos stem.

  Use Strict Arg name

  a   = 12.34e-56 + " -98.76e+123 "     -- Highlighting of numbers
  len = Length( Stem.12.2a.x.y )        -- A built-in function call
  pos = Pos( "S", "String" )            -- An internal function call
  Call External pos, len, .True         -- An external function call
  .environment~test.2.x = test.2.x      -- Method call, compound variable

  Exit "नमस्ते"G,  "P ≝ 𝔐",  "🦞🍐"     -- Unicode strings
```

And this is a plain (non-Rexx) code block, showing the YAML front
matter syntax:

```
---
rexxpub:
  docclass: rexxdoc-chapter
  chapter: 3
---

# Keyword Instructions
```

Both types of code block share the same light grey background and
8pt monospace font, matching the official ooRexx documentation.

## Note and Warning Boxes

The ooRexx documentation uses coloured boxes for notes and warnings.
The `rexxdoc-chapter` class supports these through Pandoc fenced divs.

A note box uses an olive header (`#8e9e00`):

::: note
This is a note.  In the ooRexx documentation, notes provide
additional context or remind the reader of an important rule.  The
header bar is automatically generated by CSS --- the Markdown source
only needs the `div.note` wrapper.
:::

A warning box uses a red header:

::: warning
This is a warning.  Warnings highlight potential pitfalls or
common mistakes.  The same fenced div mechanism is used, with the
class `warning` instead of `note`.
:::

The Markdown source for a note is simply:

```
::: note
Your note text here.
:::
```

## Running Headers and Page Numbers

In the print rendering (via `?print=pdf` or `md2pdf`), the page
layout replicates the official ooRexx documentation:

- **Running header:** the current section title appears at the
  top right of each page, separated from the content by a thin
  horizontal rule.  On the first page of a section, the header
  is suppressed.
- **Page number:** centred at the bottom of each page, with a
  thin rule above.
- **Margins:** asymmetric, matching the FOP output used by the
  ooRexx documentation --- approximately 31mm left and 19mm right,
  giving more space on the binding side.

## Relationship with Other Document Classes

The `rexxdoc-chapter` class is loaded *after* `rexxpub-base.css`,
just like `article`, `book`, `letter`, and `slides`.  It inherits
all the shared infrastructure: section numbering (via
`numberSections.js`), figure and listing captions (via
`numberFigures.js`), footnote support, and paged.js split-element
fixes.

The key differences from `article` are:

- Sans-serif typography (Liberation Sans) instead of Times New Roman.
- DocBook-style paragraph spacing instead of LaTeX first-line indent.
- Blue headings instead of black.
- Chapter label ("Chapter N.") with horizontal rule.
- Section numbers prefixed with the chapter number.
- Asymmetric page margins.

### Future Work

A companion `rexxdoc` class (for full multi-chapter books) is
planned but not yet implemented.  It would handle the table of
contents, parts, and cross-chapter navigation, using the same
visual style as `rexxdoc-chapter` for individual chapters.
